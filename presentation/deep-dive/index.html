Parte 1

O que muda num chart profissional?

Amador vs Pro

Chart "Ingênuo"

  • Tudo no `values.yaml` sem limite.
  • Se o valor for digitado errado (ex: string vs int), o erro estoura no Kubernetes.
  • DRY não existe. Muitos if/elses gigantes.

Chart Seguro

  • `values.schema.json` recusa values errados no CI.
  • Pedaços comuns extraídos para `_helpers.tpl`.
  • `helm test` diz se a release realmente funciona.

Parte 2

Contratos Firmes

Garantindo que a configuração está certa antes de tentar instalar.

O Guarda-costas: Schema

O arquivo `values.schema.json` aplica o padrão JSON Schema aos seus values.

  • Obriga campos essenciais.
  • Impede que enviem string onde se espera int.
  • `helm lint` e `helm template` falham rápido e blindam seu cluster.
{ "$schema": "http://json-schema.org/schema#", "type": "object", "required": ["replicaCount"], "properties": { "replicaCount": { "type": "integer", "minimum": 1 } } }

Parte 3

Helpers Avançados

Menos cópia, mais segurança.

Reuso com _helpers.tpl

O arquivo _helpers.tpl

Pedaços de template que não criam recursos no K8s por si só, mas definem blocos (como labels padronizadas) que você pode "injetar" em todo lugar.

Ação: include

Em vez de escrever a mesma anotação em 5 arquivos YAML, você escreve `{{ include "meuchart.labels" . }}`.

Parte 4

Validação Pós-Deploy

Não basta deployar; tem que garantir que a aplicação atende a porta.

Helm Test

Seu CI deployou no ambiente. A release falhou?

  • O Helm pode rodar "Test Pods" (arquivos no `templates/tests/`).
  • Um teste clássico é subir um pod de Alpine que tenta fazer um `wget` no seu Service.
  • Se passar: Sucesso! Se falhar: Aciona o seu alerta ou CI quebra.
helm test myapp -n apps
