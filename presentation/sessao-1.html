Parte 1

Por que Helm existe?

Da dor do YAML repetido ao pacote versionado com histórico.

A dor antes do Helm

Complexidade operacional

  • Muito YAML repetido para ambientes diferentes.
  • Dificuldade de rastrear o que entrou no deploy.
  • Rollback manual e inseguro.

Impacto no time

  • Pouca clareza sobre qual versão está rodando.
  • Deploy vira coleção de objetos soltos.
  • Aplicar manifest errado no cluster custa mais caro.
Kubernetes entende objetos. Helm ajuda a tratar aplicações como pacotes versionados.

O que Helm é e o que não é

✅

Helm é

  • Package manager para Kubernetes.
  • Instala uma release versionada.
  • Fluxo previsível entre chart, values e manifests.
🛑

Helm não é

  • Substituto de Kubernetes.
  • Substituto de GitOps ou observabilidade.
  • Correção mágica para chart mal desenhado.
chart + values -> render -> manifests -> release

Conceitos-base

Chart
Pacote com templates, values e metadados.
Release
Instância instalada de um chart.
Values
Configuração que alimenta os templates.
Template
YAML Kubernetes com Go Template.

Parte 2

O que existe dentro de um chart?

Estrutura mínima para operar, configurar e validar.

Anatomia mínima de chart

workshop-app/ Chart.yaml values.yaml templates/ deployment.yaml service.yaml
Chart.yaml

Identidade e versão do pacote.

values.yaml

Contrato de configuração.

templates/

Recursos Kubernetes renderizados.

Contrato vs implementação

Contrato

`values.yaml` define o que o usuário pode mudar e quais parâmetros são esperados.

Implementação

`templates/` traduz configuração em objetos Kubernetes prontos para o cluster.

Se todo mundo precisa editar template, o contrato do chart está fraco.

Parte 3

Operação mínima

O fluxo que reduz risco e deixa rollback sempre ao alcance.

Antes de aplicar no cluster

helm lint ./chart helm template myapp ./chart -f values.yaml
lint

Valida a estrutura básica do chart.

template

Mostra o YAML final antes de tocar no cluster.

Operando a release

helm upgrade --install myapp ./chart -n apps --create-namespace helm history myapp -n apps helm rollback myapp 1 -n apps

upgrade --install

Padrão mais seguro para automação e idempotência.

history

Linha do tempo da release para auditoria e diagnóstico.

rollback

Retorno controlado para uma revision anterior.

O ciclo mínimo profissional

1. lint

Valida estrutura e problemas básicos.

2. template

Revisa o YAML final antes do apply.

3. upgrade --install

Entrega de forma reproduzível.

4. history

Audita a linha do tempo da release.

5. rollback

Recupera rápido quando necessário.

Erros comuns de iniciante

Pular helm template e descobrir erro só no cluster.

Usar --set demais em produção.

Não saber explicar o que está em values.yaml.

Não versionar mudança de chart.

Tratar rollback como último recurso, e não como parte do fluxo.

Confundir pacote reutilizável com manifest avulso.
